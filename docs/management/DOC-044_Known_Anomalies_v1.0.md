## 1. 목적

본 문서는 HnVue Console SW (RadiConsole™)의 허가 제출 시점 또는 각 릴리즈 시점에서 알려져 있지만 해결되지 않은 SW 결함(버그, 이상 동작, 기능 제한)을 목록화한다.

각 결함의 임상적·안전적 영향과 허용 근거를 명시하여, 규제 기관이 잔여 위험을 평가할 수 있도록 투명하게 공개한다.

**알려진 결함이 없는 경우에도** "현재 알려진 결함 없음 (No Known Anomalies)"을 명시적으로 기재하고 QA 책임자가 서명하여 제출한다.

본 문서는 다음 규격 요건을 충족한다:

- **IEC 62304:2006+AMD1:2015** §6.2.3 (결함 수정 절차), §9.8 (Revision History and Anomaly List)
- **FDA SW Guidance (2019)** Section IV-J (Known Anomalies / Defects)
- **MFDS 안내서-1425-01** SW 이력 관리 — 잔여 결함 공개 요건
- **EU MDR Annex I §17.2** 소프트웨어의 알려진 한계 공개

## 4. 본문

### 4.1 결함 분류 기준

#### 4.1.1 심각도 분류

| 심각도 | 정의 | RadiConsole™ 적용 예시 |
|--------|------|----------------------|
| **Critical** | 환자 안전에 직접 위해를 가할 수 있는 결함 | 선량(DAP) 수치 오표시, 영상 환자 ID 오매핑, 촬영 파라미터 오전송 |
| **Major** | 핵심 기능 사용 불가 또는 심각한 성능 저하 | DICOM C-STORE 전송 실패, PACS 연결 불가, Worklist 조회 불가 |
| **Moderate** | 기능 일부 제한, 대안(Workaround) 존재 | 특정 DICOM 태그 누락 시 UI 경고 미표시, 특정 조건에서 재촬영 버튼 비활성화 |
| **Minor** | 기능에 영향 없는 외관상 결함, 경미한 불편 | UI 텍스트 오타, 영상 뷰어 스크롤 지연, 비기능 버튼 정렬 문제 |

#### 4.1.2 허용 기준 (Acceptance Criteria for Unresolved Anomalies)

결함이 아래 조건을 **모두** 충족하는 경우에만 미해결 상태로 릴리즈가 가능하다:

1. 환자 안전(Safety)에 직접 또는 간접 위해가 없다.
2. 진단 정확도(영상 품질, 측정값 정확성)에 영향을 미치지 않는다.
3. ISO 14971 기반 위험-편익 분석에서 잔여 위험이 허용 기준 이내이다 (위험관리 파일 참조).
4. QA 책임자 및 RA 담당자가 서면으로 허용을 승인한다 (§4.4 참조).
5. IFU(사용 설명서)에 해당 제한 사항 및 완화 조치가 기재된다 (EU MDR Annex I §17.2).

> **Critical 결함**: Critical 심각도 결함은 어떠한 경우에도 허용할 수 없으며, 해결 전까지 릴리즈가 절대 금지된다.

### 4.5 잔여 결함 위험 평가

잔여 결함의 위험 수준 평가는 ISO 14971:2019 기반 위험 관리 파일(위험관리 파일 — 작성 필요)의 잔여 위험 허용 기준을 적용한다.

**평가 방법**:
- 발생 확률(Probability of Occurrence) × 피해 심각도(Severity of Harm) = 위험 수준(Risk Level)
- 위험 수준이 허용 기준을 초과하는 결함은 릴리즈 전 반드시 해결

| 결함 ID | 발생 확률 | 피해 심각도 | 위험 수준 | 허용 기준 이내 여부 | 근거 |
|---------|----------|-----------|----------|--------------------|------|
| (해당 없음 — 개발 착수 전) | — | — | — | — | 개발 완료 후 시험 결과 기반 기재 |

> **개발 완료 후**: 발견된 각 결함에 대해 ISO 14971 위험 평가 매트릭스를 적용하여 위험 수준을 산정하고 본 표를 업데이트한다.

## 5. 비고: 이 문서가 필요한 이유

### 이 문서가 없으면?

FDA는 "알려진 결함이 없다"는 주장을 믿지 않는다. 알려진 결함 목록이 없으면 "검토하지 않은 것"으로 간주되며, 오히려 숨기고 있다는 의심을 받는다. FDA 심사관이 eSTAR를 검토할 때 Known Anomalies 목록이 없으면 즉시 AI(Additional Information) 요청을 보낸다.

반대로, 경미한 결함이 수십 개 있어도 각각에 대해 안전 영향 평가와 완화 조치를 기술하면 허가에 문제가 없다. **투명성이 신뢰를 만든다.**

- **FDA 510(k)**: "Revision History and Anomaly List"는 IEC 62304 §9.8과 FDA SW Documentation Guidance의 필수 구성 요소다. 없으면 eSTAR Section 12 미충족으로 AI 요청이 확실히 발송된다. 결함이 없더라도 "현재 알려진 결함 없음 (No Known Anomalies)"을 명시적으로 기재해야 한다.
- **MFDS 2등급**: 소프트웨어 이력 관리 요건 하에 잔여 결함 공개를 요구한다. 결함 목록이 없으면 "소프트웨어 검증이 충분히 이루어지지 않았다"는 판단의 근거가 된다.
- **EU MDR Class IIa**: Annex I §17.2는 "소프트웨어의 알려진 한계"를 공개하도록 요구한다. 결함 목록 미공개는 IFU(사용 설명서)의 금기 사항·경고 기재 의무와도 연결되어 부적합 판정을 초래한다.

### 시장별 요구 수준

| 시장 | 요구 수준 | 설명 |
|------|----------|------|
| FDA 510(k) | 필수 | eSTAR Section 12 필수 항목. 없으면 AI 요청 확실. "결함 없음"도 명시적 기재 필요 |
| MFDS 2등급 | 필수 | SW 이력 관리 요건. 미공개 시 검증 불충분 판단 근거가 됨 |
| EU MDR Class IIa | 필수 | Annex I §17.2 직접 요건. IFU 경고 문구와 연계. 미공개 시 부적합 판정 |

### 이 문서의 작성·유지 원칙

1. **사전 수립**: 개발 착수 전 프레임워크 및 분류 기준을 수립한다 (본 v1.0 문서의 목적).
2. **실시간 업데이트**: 시험 과정에서 결함 발견 즉시 기재한다.
3. **릴리즈 전 재확인**: 릴리즈 승인 전 전체 결함 목록을 검토하고 승인 서명을 갱신한다.
4. **투명성 유지**: 해결된 결함도 §4.7에 이력을 남긴다.
5. **IFU 연계**: 허용된 결함의 완화 조치는 반드시 IFU(사용 설명서)에 반영한다.
