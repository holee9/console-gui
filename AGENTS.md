# Codex 운영 지침

이 문서는 Codex가 이 저장소에서 작업할 때 우선 참조하는 간결한 운영 지침이다.
`.moai/`는 참고 자료로만 사용하며, 사용자가 명시적으로 요청하지 않는 한 수정하지 않는다.

## 기본 원칙

- 먼저 실행하고, 끝나면 검증과 결과를 짧게 보고한다.
- 같은 정보를 반복해서 읽지 않는다.
- 필요한 파일만 좁게 읽고, 광범위한 덤프는 피한다.
- 로컬 코드와 문서를 우선 참조하고, 외부 탐색은 꼭 필요할 때만 한다.
- 사용자 변경과 내 변경을 섞지 않는다.

## 컨텍스트 절약

- 한 번에 필요한 파일만 읽는다. 기본 목표는 `8`개 이하 파일, 파일당 `250`줄 이하다.
- 검색은 `rg`를 우선 사용한다.
- 이미 읽은 내용은 요약을 재사용하고, 다음 경우에만 다시 연다.
  - 수정 직전
  - 파일이 바뀌었을 가능성이 있을 때
  - 검증 근거가 새로 필요할 때
- 불필요한 `git status`, 전체 파일 출력, 장문 계획 반복을 피한다.

## 실행 규칙

- 명확한 작업은 바로 수행한다.
- 아래 경우에만 질문한다.
  - 파괴적이거나 되돌리기 어려운 작업
  - 제품 결정에 비자명한 선택지가 여러 개 있을 때
  - 자격증명, 외부 접근, 배포 권한이 필요할 때
  - 안전/정책 충돌이 있을 때
- `.moai`는 읽기 전용 참고 영역으로 간주한다.

## 자동승인 정책

다음 작업은 Codex가 자동 승인으로 진행한다.

- 문서만 수정하는 작업
- `.md`, `.json`, `.yaml`만 수정하고 런타임 영향이 없는 작업
- 읽기 전용 분석, 리뷰, 보고 작업
- `5`개 이하 파일 변경이고 공개 API, 워크플로우 상태, 보안 동작을 건드리지 않는 작업

다음 작업은 사용자 확인이 필요하다.

- 인증, 인가, 시크릿, 감사 로그 처리 변경
- 데이터베이스 스키마 또는 데이터 마이그레이션
- 패키지, SDK, 런타임 버전 업그레이드
- 공개 인터페이스, 계약, 워크플로우 상태 전이 변경
- 강제 푸시, 원격 변경, 브랜치 전략 변경
- 재귀 삭제, 대량 이동, 복구 곤란 작업

## 검증 기준

- 코드 변경 시 영향받는 프로젝트를 빌드한다.
- 가능하면 영향받는 테스트만 우선 실행하고, 필요 시 솔루션 테스트로 넓힌다.
- 문서 전용 변경은 링크, 경로, 명령, 인코딩을 확인한다.
- 커밋 전에는 내가 수정한 파일만 스테이징한다.

## 저장소 정합성

- 이 저장소의 기준 스택은 `.NET 8`, `WPF`, `CommunityToolkit.Mvvm`, `EF Core`, `xUnit`이다.
- 아키텍처 핵심 규칙:
  - `HnVue.App`는 Composition Root다.
  - `HnVue.Common`은 하위 공통 계층이다.
  - `HnVue.UI`, `HnVue.UI.Contracts`, `HnVue.UI.ViewModels` 분리를 유지한다.
  - `Result` / `Result<T>` 기반 오류 흐름을 우선 존중한다.

## Git 규율

- 관련 파일만 스테이징한다.
- 순차적 Git 명령을 선호한다.
- 사용자가 만든 다른 변경은 되돌리지 않는다.
- 요청이 없으면 자동 커밋, 자동 푸시, 자동 브랜치 변경을 하지 않는다.
