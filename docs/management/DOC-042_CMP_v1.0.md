## 1. 목적

본 문서는 HnVue Console SW (HnVue) 및 모든 관련 산출물의 버전 관리, 변경 제어, 베이스라인 설정, 추적성을 보장하는 형상 관리 프로세스를 정의한다.

"언제든 FDA 510(k) 제출 버전의 SW를 동일하게 재현할 수 있음"을 증명하기 위한 핵심 프로세스 문서로, 허가 후 감사(GMP Inspection) 대응의 근거가 된다.

본 문서는 다음 규격 요건을 충족한다:

- **IEC 62304:2006+AMD1:2015** §8.1 (SW 형상관리 수립), §8.3 (변경 통제)
- **FDA SW Guidance (2019)** Section IV-G (형상관리)
- **ISO 13485:2016** §4.2.3 (문서 관리), §7.3.8 (설계 및 개발 변경 관리)
- **MFDS 안내서-1425-01** SW 이력 관리 요건

| CI-002 | 설계 문서 (SDP, SRS, SAD 등) | 문서 | 문서 저장소 (docs/) | 문서번호(DOC-XXX) + 버전(v[X.X]) |

### 4.3 버전 관리 정책 (VCS Policy)

#### 4.3.1 버전 번호 체계

HnVue은 Semantic Versioning 2.0.0 (semver.org) 기반 버전 번호 체계를 따른다.

```
MAJOR.MINOR.PATCH
│      │     └── 패치 버전: 하위 호환 버그 수정
│      └──────── 부 버전: 하위 호환 기능 추가
└─────────────── 주 버전: 비호환 변경 / 주요 기능 개편
```

| 구분 | 예시 | 설명 |
|------|------|------|
| 초기 릴리즈 | `v1.0.0` | Phase 1 최초 인허가 버전 |
| 부 기능 추가 | `v1.1.0` | Phase 2 AI 기능 추가 등 |
| 버그 수정 | `v1.0.1` | 비안전 관련 결함 패치 |
| 주요 개편 | `v2.0.0` | 아키텍처 변경, 비호환 인터페이스 변경 |

- **Pre-release**: `v1.0.0-alpha.1`, `v1.0.0-rc.1` 형식 사용
- **빌드 메타데이터**: `v1.0.0+build.20260901` 형식 사용 (릴리즈 기록 추적용)

#### 4.3.2 브랜칭 전략 (Git Flow 기반)

| 브랜치 | 목적 | 보호 설정 | 병합 대상 |
|--------|------|----------|----------|
| `main` | 릴리즈 완료 코드 (허가 버전) | 직접 Push 금지, PR + 2인 승인 필수 | `release/*`에서만 병합 |
| `develop` | 개발 통합 브랜치 | PR + 1인 승인 | `feature/*`에서 병합 |
| `feature/[기능명]` | 신규 기능 개발 (예: `feature/dicom-c-store`) | 없음 | `develop`으로 병합 |
| `hotfix/[이슈ID]` | 긴급 버그/보안 수정 | 없음 | `main` 및 `develop`으로 병합 |
| `release/[버전]` | 릴리즈 준비 (최종 검증, 버전 업데이트) | PR + 2인 승인 | `main` 및 `develop`으로 병합 |

#### 4.3.3 커밋 메시지 규칙

```
[유형]: [요약] (#이슈번호)

유형:
  feat     — 신규 기능 (IEC 62304 §5.5 구현)
  fix      — 버그 수정 (IEC 62304 §6.2.3 결함 해결)
  docs     — 문서 변경
  test     — 시험 코드 추가/수정 (xUnit)
  refactor — 기능 변경 없는 코드 리팩토링
  chore    — 빌드 설정, 의존성 업데이트
  security — 보안 취약점 패치

예:
  feat: DICOM C-STORE TLS 1.3 지원 추가 (#123)
  fix: fo-dicom 5.1 영상 표시 타임아웃 오류 수정 (#456)
  chore: SQLite 3.45.1 → 3.46.0 업그레이드 (CVE 패치) (#789)
```

#### 4.3.4 릴리즈 태그 정책

- **형식**: `v[MAJOR.MINOR.PATCH]` (예: `v1.0.0`)
- **태그 생성 시점**: `release/*` 브랜치가 `main`에 병합된 후 즉시 생성
- **태그 서명**: GPG 서명 적용 (권장) 또는 SHA-256 해시 기록 (대안)
- **태그 삭제**: 절대 금지 — 과거 릴리즈 버전 접근 가능성 보장 필요
- **태그 설명 (Annotated Tag)**: 릴리즈 요약, 빌드 해시, 담당자 포함

### 4.4 빌드 프로세스

#### 4.4.1 CI/CD 파이프라인 개요

HnVue 빌드 파이프라인은 재현 가능한 빌드(Reproducible Build)를 보장하기 위해 고정된 빌드 환경에서 실행된다. (상세: DOC-043)

```
[코드 Push / PR 생성]
    → [정적 분석] (Roslyn Analyzers / SonarQube — 작성 필요)
    → [단위 테스트 실행] (xUnit 2.7.0, Coverlet 6.0.0)
    → [빌드] (MSBuild + .NET 8 SDK, Release/x64)
    → [통합 테스트] (해당 시)
    → [빌드 산출물 SHA-256 해시 생성]
    → [코드서명] (MSI — signtool.exe)
    → [아티팩트 저장소 업로드]
```

> **빌드 환경 상세**: DOC-043 (소스코드 및 빌드 환경 기록) §4.3 참조

#### 4.4.2 릴리즈 빌드 절차

1. `release/[버전]` 브랜치 생성 (`develop`에서 분기)
2. 버전 번호 업데이트 (AssemblyInfo.cs / Directory.Build.props, MSI 버전, 라벨 텍스트)
3. 최종 버그 수정 및 코드 리뷰 (PR + 2인 승인)
4. 릴리즈 빌드 실행 (재현 가능한 빌드 환경 — DOC-043 참조)
5. 빌드 산출물 SHA-256 해시 생성 및 DOC-043에 기록
6. 코드서명 적용 (MSI 설치 패키지 — EV Code Signing 권장)
7. QA 시험 통과 확인 (단위/통합/시스템 시험 전체 Pass)
8. Known Anomalies 확인 및 DOC-044 업데이트
9. `main` 병합 및 릴리즈 태그 생성 (`v[버전]`)
10. 릴리즈 기록(DOC-034) 생성 및 승인

#### 4.4.3 빌드 재현성 보장

- 빌드 환경은 [Docker 이미지 또는 VM 스냅샷 — 작성 필요]으로 고정 (상세: DOC-043 §4.5)
- 모든 NuGet 의존성은 `packages.lock.json`으로 버전 고정 (`--locked-mode` 빌드)
- 빌드 환경 변경 시 새 이미지/스냅샷 생성 및 이전 이미지 보존 (DOC-043 §4.7)
- CI/CD 파이프라인 로그를 아티팩트로 보관 (릴리즈 태그와 연계)

### 4.5 변경 통제 프로세스 (Change Control)

#### 4.5.1 변경 요청 (Change Request, CR) 절차

```
1. CR 생성
   - 이슈 추적 시스템(GitHub Issues / Jira)에 CR 티켓 생성
   - CR 유형: Bug Fix / Feature / Security Patch / Documentation / SOUP Update

2. 영향도 평가 (IEC 62304 §6.2.5 준수)
   - 기능 영향: 어느 SW 아이템(컴포넌트)에 영향을 미치는가?
   - 안전 영향: 환자 안전 관련 기능(선량 표시, 영상 정확도 등)에 영향이 있는가?
   - 규제 영향: FDA 510(k) 변경보고 / MFDS 변경신고 / EU MDR 재평가 필요 여부
   - SOUP 영향: SOUP 버전 변경 필요 여부 (SBOM 업데이트 필요)

3. 승인
   - Minor 변경: SW 개발 책임자 승인
   - Major 변경: PM + QA 책임자 승인
   - Safety-related 변경: PM + QA + RA 책임자 승인

4. 구현 및 시험
   - 구현 (feature/* 또는 hotfix/* 브랜치)
   - 코드 리뷰 + 회귀 시험 (xUnit 단위/통합 테스트 재실행)

5. 베이스라인 업데이트
   - 관련 문서 개정 (SRS, SAD, 위험관리 등)
   - DOC-019 (SBOM) SOUP 버전 업데이트 (해당 시)
   - DOC-042 (본 CMP) 및 DOC-043 빌드 환경 기록 업데이트 (해당 시)
```

#### 4.5.2 변경 유형별 승인 권한

| 변경 유형 | 정의 | 승인 권한 | 규제 보고 |
|----------|------|----------|----------|
| Minor | 기능 영향 없는 버그 수정, UI 텍스트, 비기능 개선 | SW 개발 책임자 | 내부 기록만 |
| Major | 새 기능, 성능 변경, SOUP 버전 업그레이드, 아키텍처 변경 | PM + QA 책임자 | MFDS 경미한 변경 신고 검토 |
| Critical (Safety) | 선량 계산, 영상 정확도, 환자 ID 연계 등 안전 관련 기능 변경 | PM + QA + RA + 대표이사 | FDA/MFDS 변경허가 검토, EU MDR 재평가 |
| Emergency (Hotfix) | 즉시 패치 필요한 보안/안전 취약점 (CVE Critical 등) | QA + RA (사후 PM 보고) | 규제 보고 여부 24시간 내 판단 |

> **SOUP 버전 변경**: IEC 62304 §8.3 요건에 따라 별도의 SOUP 변경 평가를 수행하고 DOC-019 (SBOM)를 업데이트한다.

### 4.7 형상 상태 관리

#### 4.7.1 문서 상태 코드

| 상태 | 설명 | 다음 상태 |
|------|------|----------|
| Draft | 초안 작성 중 | Review |
| Review | 검토 중 | Approved 또는 Draft (재작성) |
| Approved | 공식 승인 완료 (베이스라인 포함) | Obsolete 또는 Draft (개정 시) |
| Obsolete | 상위 버전으로 대체됨 | — |

#### 4.7.2 형상 상태 보고

- **보고 주기**: 월 1회 (또는 마일스톤 시)
- **보고 내용**: 각 CI의 현재 버전, 상태(Draft/Review/Approved), 해당 기간 변경 내역
- **보고 대상**: PM, QA 책임자, RA 담당자
- **보고 형식**: 이슈 추적 시스템 대시보드 + 월간 형상 상태 요약 이메일

#### 4.7.3 형상 감사 (Configuration Audit)

| 감사 유형 | 시점 | 목적 | 담당 |
|----------|------|------|------|
| 기능 형상 감사 (FCA, Functional Configuration Audit) | 릴리즈 전 | SW 기능이 SRS 요구사항을 충족하는지 확인 | QA 책임자 |
| 물리적 형상 감사 (PCA, Physical Configuration Audit) | 릴리즈 전 | 실제 산출물이 형상 문서(버전, 해시 등)와 일치하는지 확인 | QA 책임자 |
| 내부 감사 | 분기 1회 | 형상관리 프로세스 준수 여부 확인 (IEC 62304 §9.7) | QA 책임자 |
| 규제 대응 감사 | FDA/MFDS/인증기관 요청 시 | 외부 감사 대응 | RA 담당자 + QA 책임자 |

## 5. 비고: 이 문서가 필요한 이유

### 이 문서가 없으면?

형상관리 계획(CMP)이 없으면 "어떤 버전을 테스트했는가", "FDA에 제출한 버전이 실제 출하 버전과 같은가"를 증명할 방법이 없다. 이는 모든 V&V 결과를 무효화하는 치명적 결함이다.

- **FDA**: 허가 후 GMP Inspection(현장 감사)에서 형상관리 기록 부재는 Quality System 전반의 신뢰성을 의심받는 사유가 된다. SW 버전과 테스트 결과의 연결이 증명되지 않으면 모든 V&V 결과가 무효화될 위험이 있으며 Warning Letter 발행 사례가 있다.
- **MFDS**: 변경 허가·신고 시 이전 버전과의 비교를 요구한다. 형상관리 기록 없이는 버전 간 변경 사항을 객관적으로 제시할 수 없다.
- **EU MDR**: ISO 13485 §4.2.3(문서 관리), §7.3.8(설계 및 개발 변경 관리)에서 형상관리를 QMS의 필수 요소로 규정한다. 형상관리 계획 없이는 EU MDR Class IIa 인증 유지가 불가능하다.

### 시장별 요구 수준

| 시장 | 요구 수준 | 설명 |
|------|----------|------|
| FDA 510(k) | 필수 | GMP Inspection 필수 확인 항목. 부재 시 Warning Letter 발행 가능. 모든 V&V 결과 무효화 위험 |
| MFDS 2등급 | 필수 | 변경 허가·신고 시 버전 비교 근거 문서. 형상관리 없으면 변경 신고 자체가 불가 |
| EU MDR Class IIa | 필수 | ISO 13485 §4.2.3, §7.3.8 직접 요건. 형상관리 없이 ISO 13485 인증 유지 불가 |
