### 1.2 PMS 목표

EU MDR Article 83에 따라 다음 목표를 달성하기 위한 PMS 시스템을 수립한다:

1. HnVue Console SW의 안전성, 성능 및 허용 가능한 품질 수준을 CE 인증 유지 기간 동안 지속적으로 확인
2. 잔여 위험 수용 가능성 지속 검토 (DOC-008 RMP §8 잔여 위험 평가 연계)
3. CER (DOC-029)을 갱신하기 위한 임상 데이터 지속 수집
4. 이상사례 및 FSCA(Field Safety Corrective Action)를 조기에 식별하고 규정 기한 내에 적절히 대응
5. 사이버보안 취약점 발생 시 신속한 패치 개발·배포 및 사용자 통지 (DOC-048 VMP 연계)
6. PMCF 계획(파트 2)에서 정의된 임상 질문에 대한 데이터를 수집하여 잔여 임상 불확실성 해소

### 1.4 이상사례 보고 기준 및 절차

#### 1.4.1 EU MDR Article 87 보고 기한

| 이상사례 유형 | 보고 기한 | 보고 대상 |
|------------|---------|---------|
| 공중 보건 심각한 위협 (Serious Public Health Threat) | 2일 이내 | 각 EU 회원국 관할 기관 + EUDAMED |
| 사망 또는 예상치 못한 심각한 위해 (Death or Unanticipated Serious Deterioration) | 10일 이내 | 각 EU 회원국 관할 기관 |
| 심각한 공중 위해 위험 외 이상사례 | 15일 이내 | 각 EU 회원국 관할 기관 |
| 추세 보고 (Trend Reporting — Article 88) | 30일 이내 | 각 EU 회원국 관할 기관 |
| FSCA 사전 고지 | FSCA 시행 전 (가능한 경우) | EUDAMED + 각 회원국 기관 |

#### 1.4.2 이상사례 처리 절차

```
[접수]
고객/사용자/규제 기관으로부터 이상사례 접수
    ↓
[초기 평가 — 24시간 이내]
이상사례 해당 여부 판단 (EU MDR Article 2(64) 정의 기준)
심각도 예비 평가 (Serious / Non-serious)
    ↓
[심각도 분류 — 48시간 이내]
EU MDR Article 87 기준 보고 기한 확정
    ↓
[규제 기관 보고 — 규정 기한 내]
EUDAMED 또는 각 회원국 규제 기관에 보고
    ↓
[조사 및 근본 원인 분석]
SW 오류 로그 분석, FMEA 검토, 위험 관리 파일 검토
    ↓
[CAPA 수립 및 실행]
수정 조치 (Corrective Action) + 예방 조치 (Preventive Action)
FSCA 필요 여부 평가
    ↓
[DOC-008 위험 관리 파일 업데이트]
위험 등록부(Risk Register), FMEA 갱신
    ↓
[DOC-029 CER 갱신 여부 검토]
새로운 임상적 증거 발생 시 CER 즉시 갱신
```

#### 1.4.3 SW 특화 이상사례 판단 기준

| SW 오작동 유형 | 이상사례 해당 여부 | 판단 근거 |
|--------------|-----------------|---------|
| DICOM 영상 전송 실패 (환자 진단 지연) | ⚠️ 평가 필요 | 진단 지연이 환자 위해로 이어졌는지 여부 확인 |
| 영상 표시 오류 (잘못된 W/L로 판독 오류 가능) | ✅ 이상사례 해당 가능 | EU MDR Article 87 기준: 의료기기 오작동으로 인한 심각한 위해 가능성 |
| 환자 정보 불일치 (Wrong Patient 오류) | ✅ 이상사례 해당 | 잘못된 환자에게 방사선 피폭 위험 |
| SW 크래시 (촬영 중 강제 종료) | ⚠️ 평가 필요 | 임상 작업 중단 여부, 환자 위해 가능성 평가 |
| 사이버보안 침해 (Critical 취약점 익스플로잇) | ✅ FSCA 해당 가능 | 섹션 1.5 참조 |

### 1.5 FSCA 및 사이버보안 패치 연계

#### 1.5.1 사이버보안 취약점 FSCA 판단 매트릭스

DOC-016 (CMP), DOC-048 (VMP), DOC-049 (IEC 81001-5-1 §6.2–6.3)과 연계하여 사이버보안 취약점 발생 시 FSCA 여부를 판단한다.

| 취약점 등급 | CVSS 범위 | 임상 영향 가능성 | FSCA 해당 여부 | 배포 SLA | FDA §524B 대응 |
|-----------|---------|--------------|--------------|---------|--------------|
| **Critical** | ≥ 9.0 | 높음 — 영상 변조, 무단 접근, 서비스 거부 가능 | ✅ FSCA 필요 (임상 영향 평가 후 최종 결정) | 패치 배포 ≤ 14일 | 즉시 보고 |
| **High** | 7.0–8.9 | 중간 — 인증 우회, 권한 상승 가능 | ⚠️ 임상 위험 평가 후 결정 | 패치 배포 ≤ 30일 | 30일 내 조치 계획 |
| **Medium** | 4.0–6.9 | 낮음 — 제한적 악용 가능 | ❌ 일반적으로 FSCA 불필요 | 패치 배포 ≤ 90일 | 정기 업데이트 |
| **Low** | < 4.0 | 최소 | ❌ FSCA 불필요 | 다음 정기 릴리즈 | 릴리즈 노트 |

#### 1.5.2 FSCA가 필요한 사이버보안 사고 예시

| 사고 유형 | FSCA 해당 여부 | 조치 방향 |
|---------|--------------|---------|
| 영상 변조를 유발할 수 있는 Critical 취약점 발견 및 익스플로잇 확인 | ✅ FSCA 필요 | 즉시 패치 배포 + 사용자 긴급 통지 + 규제 기관 보고 (2일 이내) |
| 인증 우회 취약점 (CVSS ≥ 7.0) — 임상 영향 미확인 | ⚠️ 임상 위험 평가 후 결정 | 위험 평가 → FSCA 여부 결정 → 패치 배포 (30일 이내) |
| SOUP 라이브러리 취약점 (VEX: Not Exploitable 판정) | ❌ FSCA 불필요 | DOC-046 VEX 업데이트, 다음 정기 패치 포함 |
| 무단 접근 시도 탐지 (실제 침해 없음) | ❌ FSCA 불필요 | 보안 로그 보존, 감사 로그 분석, 필요 시 위험 재평가 |

#### 1.5.3 SW 특화 FSCA 조치 유형

| 조치 유형 | 내용 | 해당 상황 |
|---------|------|---------|
| 긴급 강제 업데이트 | 다음 기동 시 자동 업데이트 강제 (사용자 선택 불가) | Class I 리콜 수준 Critical 취약점 (임상 영향 확인된 경우) |

| 임시 기능 비활성화 | 취약한 특정 기능을 원격 구성으로 비활성화 | 패치 개발 완료 전 임시 조치 |

### 2.4 PMCF 방법론

| 방법 | 세부 내용 | 데이터 수집 주체 | 윤리 승인 필요 여부 |
|------|---------|--------------|----------------|
| **고객 불만 데이터 체계 분석** | 접수된 모든 불만·이상사례를 PMCF-Q 항목별로 분류 및 임상 관련성 분석 | QA/RA 담당 | 불필요 (익명화 집계 데이터) |
| **SW 오류 로그 분석** | 익명화된 오류 로그 패턴 분석 (고객 PMCF 데이터 동의 계약 필수) | 개발/QA 담당 | 고객사 계약 필요 (IRB 검토 불필요) |
| **표준화 사용자 설문조사** | 연 1회; SUS(System Usability Scale), NPS(Net Promoter Score), PMCF-Q별 구조화 문항; 최소 30명 이상 응답 목표 | QA/RA 담당 | 고객사 IRB 검토 필요 여부 확인 (의료기관별 상이) |
| **문헌 검색 갱신 (Post-Market)** | PubMed, EMBASE 반기 업데이트 검색 (DOC-029 CER §4.2 동일 프로토콜 적용); DR 콘솔 SW 관련 신규 안전성·성능 데이터 포함 | RA/임상 담당 | 불필요 |
| **기술 지원 이력 분석** | 고객 지원 기록(티켓)을 호환성·사용성·사이버보안 이슈 분류 관점으로 분석 | QA/서비스팀 | 불필요 (익명화) |
| **선량 기록 분석** | DICOM RDSR 데이터 익명화 분석 (고객 동의 시); DRL 초과 빈도 및 경고 기능 효과 평가 | QA/임상 담당 | 고객 PMCF 데이터 계약 + IRB 검토 고려 |

> **Registry-based PMCF 채택 권장**: 별도 임상 시험 대신, 실제 고객(병원) 사용 데이터를 익명화하여 수집하는 방식. 비용 효율적이며 MDCG 2020-8에서 허용하는 방법론. 고객과 **PMCF 데이터 활용 동의 계약(PMCF Data Sharing Agreement)**을 판매 계약 체결 시점부터 포함하여 법적 근거를 확보할 것.

| 분류 | 정의 | HnVue Console SW 관련 예시 |
|------|------|--------------------------|
| **Class I Recall** | 중대한 건강 위해 또는 사망 가능성 제품 | 영상 데이터 변조를 유발할 수 있는 Critical 사이버보안 취약점 (실제 익스플로잇 확인); Wrong Patient 오류로 인한 실제 방사선 과다 피폭 |
| **Class II Recall** | 일시적 또는 가역적 건강 위해 가능성 | 영상 표시 오류로 인한 판독 품질 저하를 유발하는 SW 버그; 사용자 경고 표시 오작동 |
| **Class III Recall** | 건강 위해 가능성 없음, 법규 위반 | UDI 표시 오류, IFU 오기재, EUDAMED 등록 정보 오류 |
| **Market Withdrawal** | 제조 결함 없음, 유통/행정 문제 | 라이선스 오류, 비의도적 버전 배포 |

## 5. FDA 시판후 관리 연계 (§524B 및 MDR)

### 5.1 FDA §524B 시판후 사이버보안 의무

FDA §524B (Omnibus 2022)는 Cyber Device 제조사에 대해 시판후에도 다음 사이버보안 의무를 부과한다. 본 PMS 계획과 연계하여 이행한다.

| FDA §524B 시판후 요건 | 이행 방법 | 본 문서 연계 섹션 |
|--------------------|---------|---------------|
| 사이버 취약점 지속 모니터링 | DOC-048 VMP — NVD/CISA 월 1회 조회 | 섹션 1.3 (사이버보안 취약점 모니터링) |
| 합리적 기한 내 패치 개발·배포 | DOC-048 VMP §패치 SLA (Critical ≤ 14일) | 섹션 1.5.1 FSCA 판단 매트릭스 |
| SBOM 갱신 | SW 업데이트 시마다 DOC-045 SBOM 갱신 | 섹션 1.3 (DOC-048 연계) |
| 보안 사고 FDA 보고 | 의료기기 관련 사이버보안 사고 → FDA MDR 절차 (21 CFR Part 803) | 섹션 5.2 참조 |

### 5.2 FDA MDR (Medical Device Reporting) — 21 CFR Part 803

> **참고**: FDA MDR은 EU MDR PMS와 별개의 미국 시판후 보고 의무이며, 본 문서의 주요 범위 외 사항이다. 별도 FDA MDR 내부 SOP 작성 필요.

| 항목 | 내용 |
|------|------|
| **법적 근거** | 21 CFR Part 803 (Medical Device Reporting) |
| **보고 대상** | HnVue Console SW 사용과 관련된 사망, 심각한 상해 또는 오작동(재발 시 사망·상해 가능) |
| **보고 기한** | 사망 또는 심각한 상해 초래: 30일 이내 / 즉각적 생명 위협: 5일 이내 |
| **보고 방법** | FDA MedWatch (Form 3500A) 또는 eMDR 전자 보고 |
| **내부 절차** | 별도 FDA MDR 내부 SOP 작성 필요 (본 문서 미포함) |

### 5.3 MFDS 이상사례 보고 (의료기기법 제31조)

| 항목 | 내용 |
|------|------|
| **법적 근거** | 의료기기법 제31조 (이상사례 보고 의무) |
| **보고 대상** | 사망 또는 중대한 이상사례 |
| **보고 기한** | 사망·중대 이상사례: 30일 이내 |
| **보고 방법** | 식약처 의료기기 이상사례 보고 시스템 (UADE) |
| **내부 절차** | 이상사례 접수 → 심각도 분류 → 식약처 보고 → CAPA |
